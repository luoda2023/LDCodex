## 2026-06-09

### 修复：Codex CLI `<DSML｜tool_calls>` 标签问题

**问题**：Codex CLI 0.137.0 使用 Responses API (`wire_api = "responses"`)，但桥接程序将 Chat Completions 的 `tool_calls` 转换成错误格式的 `function_call` 项（`id` + `function`），导致 Codex CLI 无法解析，将工具调用以 `<DSML｜tool_calls>` XML 标签输出。

**修复文件**：`lib/protocol/openai-responses.mjs`（`server.mjs` 实际导入的文件）

**修改内容**：

1. **`chatToResponses` 函数（非流式）**：
   - `function_call` 项格式从 `id: tc.id, function: tc.function` 
   - 改为：`call_id: tc.id, name: fn.name, arguments: fn.arguments, status: "completed"`
   
2. **流式 tool calls 处理**：
   - 修复 `delta.tool_calls` 处理逻辑，支持多个 tool calls
   - 发送正确的 `response.output_item.added` 事件（含 `call_id`, `name`, `arguments: ""`, `status: "in_progress"`）
   - 新增 `response.function_call_arguments.delta` 事件流式传输参数
   
3. **`responsesToChat` 修复**：
   - `function_call` 项：增加 null 检查 `tcId && toolName && toolArgs`
   - `function_call_output` 项：完善字段映射 `item.output || item.content`
   
4. **`body.messages` 处理**：
   - 新增对 `assistant` + `tool_calls` 消息的处理
   - 新增对 `function_call` 和 `function_call_output` 角色的处理
   - 新增对 `tool` 角色消息的支持

**影响范围**：所有使用 `/v1/responses` 路由的 API（codexAPI, hermesAPI, neizhiAPI）
**风险点**：`input_image`/`input_file` 等非标准 content block 的透传逻辑未修改，功能保留
**部署**：需同步到线上服务器（`root@47.114.75.115`）

**顺便发现**：`lib/openai-responses.mjs`（无 `protocol/` 前缀）未被任何文件导入，是遗留副本
